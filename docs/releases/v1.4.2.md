## Highlights

### 1 · Project-Local Config at `.hive/agent-hive.json` (PR #77)

Agent Hive now has an explicit, documented config resolution chain:

| Priority | Path | Trigger |
|----------|------|---------|
| 1 (highest) | `<project>/.hive/agent-hive.json` | New preferred project config |
| 2 | `<project>/.opencode/agent_hive.json` | Legacy fallback during migration |
| 3 | `~/.config/opencode/agent_hive.json` | Global fallback |

**Why it matters:** Teams with multiple repositories can now set per-repo sandbox mode, model overrides, and skill exclusions without touching the global config. The migration path is safe — if the new file is absent the old file is still honoured. If the new file is present but invalid, Hive falls back to global defaults and emits a `[hive:config]` runtime warning instead of silently using a stale state.

### 2 · `hive-helper` Is Now a Bounded Hard-Task Operational Assistant (PR #79)

`hive-helper` was previously documented as "merge-only." In practice, operators also need:
- **State clarification** after interruptions or compaction
- **Safe append-only manual follow-up tasks** when the approved DAG has room

This release formalises those three bounded modes — merge recovery, state clarification, and safe manual follow-up — while keeping the tool boundary exact (`hive_merge`, `hive_status`, `hive_context_write` only). DAG-changing requests still route back to Hive/Swarm for plan amendment.

**`helperStatus` in `hive_status()` (OpenCode-only):** A new machine-readable field surfaces interrupted wrap-up state directly:
```json
{
  "helperStatus": {
    "doneTasksWithLiveWorktrees": [...],
    "dirtyWorktrees": [...],
    "nonInProgressTasksWithWorktrees": [...],
    "manualTaskPolicy": { ... },
    "ambiguityFlags": [...]
  }
}
```
Orchestrators can now inspect stuck state programmatically instead of guessing from commit history.

### 3 · GitHub Copilot Restored as a VS Code Desktop Preview Path (PR #80)

`v1.4.0` correctly identified OpenCode as the first-class runtime and retired overreaching Copilot promises (GitHub.com, cloud, CLI, JetBrains). VS Code *desktop* was over-cut. The Copilot LM tool registration now works again and the generated artifacts have been modernised:

- `copilot-instructions.md` — concise repository-wide steering that complements `AGENTS.md`
- `*.prompt.md` files under `.github/prompts/` — reusable entry points for planning, execution, review-request, and verification
- Guidance that leans on Copilot's built-in **browser tools**, **MCP**, and **Playwright** rather than inventing extension-specific replacements

> OpenCode remains first-class. VS Code desktop is a supported preview for teams not yet ready to commit full-time.

## Breaking Changes

None. No `.hive/` data-model or task-service API changes visible to existing nests.

**Config path migration (non-breaking):** If you have a `<project>/.opencode/agent_hive.json` today, it will continue to work as a legacy fallback. To opt into the new preferred path, copy or move it to `<project>/.hive/agent-hive.json`. The two files can coexist during migration — the new file takes precedence and the old file becomes the fallback.

## Philosophy Alignment

This release is guided by three of Hive's core principles:

**P7 (Iron Laws + Hard Gates):** Config resolution is now a deterministic contract with a documented three-level chain, test coverage, and a hard warning when the chain is broken. Helper mode boundaries are locked to an exact tool set; DAG-changing requests are rejected at the API level, not just in prompt text.

**P8 (Cross-Model Prompts):** The `helperStatus` machine-readable surface and the `manualTaskPolicy` accept/reject matrix mean operators don't have to infer Hive's state from narrative output or model reasoning — the state is observable directly.

**P3 (Human Shapes, Agent Builds):** The VS Code/Copilot restoration reflects an honest answer to the shape question: "What is this for?" OpenCode is the first-class harness. VS Code desktop is a supported preview. The shipped artifact now matches the stated direction instead of either overclaiming or over-cutting.

See the [PHILOSOPHY.md § v1.4.2](../../PHILOSOPHY.md) evolution note for the full rationale.
